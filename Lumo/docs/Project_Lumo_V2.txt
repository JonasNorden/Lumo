# NY VIKTIG GIT-INSIKT — ROOT CAUSE TILL GIT-SNURRAN

Det som hände:

* lokala commits på main
* merge commits skapade lokalt
* konfliktlösningar lokalt
* testdata lokalt
* samtidigt som origin/main fortsatte röra sig

Detta gjorde att lokal main och origin/main började divergera.

## Konsekvens

Därför blev git pull inte längre en rak synk, utan:

* Git behövde göra merge commits
* terminaleditorn öppnades
* ibland blev det merge conflicts
* användaren fick den “blå textrutan”
* arbetsflödet blev stressigt och osäkert

Det viktiga:

Detta var inte användarens fel i meningen “fel kommando” eller “fel beteende” i själva PR-flödet.

Root cause var att lokal main inte längre var identisk med origin/main.

## Hur det löstes

Den verkliga fixen var att:

* avbryta pågående merge när den var fel väg
* verifiera repo-läget
* återställa lokal main till exakt origin/main
* kasta lokala commits/testspår som inte skulle leva kvar
* återfå working tree clean

## Låst regel

Målet framåt är att alltid hålla lokal main i detta läge:

* up to date with origin/main
* nothing to commit
* working tree clean

eller, om testning pågår:

* tydligt förstådd testdiff som ska kastas efter testet

## Praktisk Git-rutin

Om test skapat filer och ändringar som bara var test, kan repo återställas till clean state.

I PowerShell:

```powershell
git restore .
git clean -fd
git status
```

eller med semikolon:

```powershell
git restore .; git clean -fd; git status
```

Viktig notering:

`&&` fungerade inte i användarens PowerShell-version.

Det betyder inte Git-fel, bara PowerShell-syntax.

## BRAKE / STICKY / RAPID

Detta spår är etablerat och ska inte tappas bort.

### Viktig designklarhet

Tre olika beteendetyper måste hållas isär:

**Brake**
tvärstopp / hög friktion / ingen glidning
fokus: stopp och grepp

**Sticky**
sänkt rörelsehastighet medan man rör sig
fokus: lera / klister / seg mark / tung framfart

**Rapid**
ökad rörelsehastighet utan boost
fokus: snabb mark / snabbare underlag

Brake ska inte omdefinieras till sticky.

### Implementerat i kod

Två nya tile behavior families finns:

* `tile.solid.sticky`
* `tile.solid.rapid`

med authored parameter:

* `movementMul`

Builder/UI/pipeline uppdaterades så att:

* Sticky och Rapid finns i Asset Builder
* de har styrkeparameter i wizardn
* de sparas via tile save bridge
* de fungerar i editor och PFH visuellt
* de kan skapas som riktiga custom tiles

### Verifierat i verkligheten

Efter rätt Editor V2-path användes verifierades att:

* Sticky och Rapid finns i vänsterpanelen
* Sticky och Rapid finns i Asset Builder
* tiles kan skapas och sparas
* de syns i katalogen
* de kan placeras i editorn
* de syns korrekt i PFH med rätt sprite

### Viktig systemsanning från denna fas

När Sticky/Rapid först verkade “inte sparas” var det inte save som var trasig.

Det som bevisades:

* Node bridge loggade `Save request: tile`
* Node bridge loggade `Save OK: tile`
* `git status` visade verkliga ändringar i `data/catalog_tiles.js`
* nya png-filer skapades i `data/assets/tiles/`

Alltså fungerade save till disk.

Det verkliga problemet då var att vänsterpanelens behavior-filter dolde nya tiles tills rätt family valdes.

## VIKTIG UX-STATUS — VÄNSTERPANELENS BEHAVIOR-DROPDOWN

## STICKY/RAPID FEEL-PROBLEMET BLEV SEDAN ÅTGÄRDAT

Tidigare verkligt utfall:

både Sticky och Rapid verkade fungera men effekten kändes ryckig / sen / instabil

särskilt över korta 24x24-tiles blev effekten för kortvarig per tile.

### Kodsanning som bekräftades

I `player.js` visade koden att:

* surface behavior samplades under spelarens fötter
* surface application låg på fel ställe i flödet i förhållande till aktuell frame-rörelse
* samplingordningen kunde ge seam-instabilitet
* korta missar över tile-gräns kunde återställa till base values för snabbt

### Codex-fix

En smal runtime-fix gjordes i `Lumo/src/game/player.js`:

* center-first surface sampling under spelarens fötter
* surface tuning appliceras före horisontell acceleration varje frame
* liten surface-state hold/hysteresis dämpar enstaka seam-missar

Ingen bred physics rewrite gjordes.

### Verkligt utfall

Det verkliga testutfallet blev mycket positivt:

* Sticky känns rätt
* Rapid känns rätt

Problemet var timing/stabilitet i runtime-surface application.

Fixen träffade rätt.

### Slutsats

M17C — Tile Behavior Runtime Alignment ska nu betraktas som klar.

_playGameplaySfx(world, path, volume = 1){

Alltså är den senare split-varianten med playbackRate i hjälpfunktionen redan borttagen igen.

```js
_playOneShot(path, volume){
  if (!path) return;
  const a = new Audio(path);
  a.preload = "auto";
  a.loop = false;
  a.volume = Math.max(0, Math.min(1, volume)) * this._getSfxVolume();
  const p = a.play();
  if (p && typeof p.catch === "function") p.catch(() => {});
}
```

Alltså finns inte längre någon console-hook kvar där som förklarar tystnaden.
